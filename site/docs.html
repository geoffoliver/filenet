Installation

Filenet ships as a standalone executable — no separate Node/Bun install required.

  1. Download filenet-bun-<platform>.zip from the Releases page for your platform (linux-x64, linux-arm64, darwin-x64, darwin-arm64, windows-x64).
  2. Extract it — you'll get filenet (the executable), an out/ folder (the UI), a drizzle/migrations/ folder, and a server/ folder (the background scan, file-watcher, and hashing workers). Keep these four together.
  3. Run the executable from that folder: ./filenet

Configuration is via environment variables: PORT (UI + management API, default 3000), P2P_PORT (default: the listening port configured in Settings), DATABASE_URL (default: ./data/filenet.db, relative to wherever you run the executable from), SCAN_LOG (default unset; set to 1 to log background scan progress to the console).

Filenet checks for new releases automatically (interval configurable in Settings, default once every 24 hours — set to 0 to disable), downloads and SHA-256-verifies them in the background, and shows a Restart to update button in Settings once one's ready. Forks can point their users at their own releases by setting the "Update repository" field in Settings (owner/repo, default geoffoliver/filenet).

If the app doesn't come back up after clicking Restart to update, the swap is crash-safe: the previous version is left behind as .old siblings in the install directory rather than being deleted outright. Look for filenet.old (filenet.exe.old on Windows), out.old, drizzle/migrations.old, server/scan-worker.js.old, server/watcher-worker.js.old, and server/hash-worker.js.old next to their non-.old counterparts, and rename each one back over the original to restore the last working version.

Removing the quarantine flag

Since the executable isn't signed, both macOS and Windows tag it as downloaded from the internet and block it from running until you clear that flag.

macOS — Gatekeeper refuses to run it with an "unidentified developer" warning. Clear the quarantine attribute from the terminal: 

xattr -d com.apple.quarantine ./filenet

If you extracted the zip with Finder rather than unzip, clear the whole folder instead, since Finder applies the attribute to every extracted file: 

xattr -dr com.apple.quarantine ./filenet-bun-darwin-arm64/

Windows — SMB/downloads mark the zip (and everything extracted from it) with a "Mark of the Web", which triggers a SmartScreen prompt. Either click More info → Run anyway on that prompt, or clear the flag first in PowerShell: 

Get-ChildItem -Path .\filenet-bun-windows-x64 -Recurse | Unblock-File

Linux has no equivalent quarantine mechanism — extracted binaries run as soon as they're marked executable (chmod +x filenet).

Running

Open http://localhost:3000. On first launch, the setup wizard walks you through configuration.

The app runs two listeners in a single process:

Listener Default Purpose
UI + API :3000 Static web UI and management API (same process)
P2P server :7734 Encrypted WebSocket connections to peers

Building from source

If you'd rather build it yourself, Filenet requires Bun ≥ 1.0: 

git clone https://github.com/geoffoliver/filenet.git
cd filenet
bun install
bun run build
bun run server

Configuration

All settings live in the app's Settings page:

  • Name — your display name, shared with friends
  • Shared folders — directories that get indexed and shared
  • Download folder — where downloaded files land
  • Auto-accept — accept friend requests from anyone, or require a password
  • Invite password — peers who supply this are auto-accepted
  • Rescan interval — how often shared folders are re-indexed (0 = manual only)
  • Port — the port peers connect to (default 7734); you must forward this on your router
  • Auto-open browser — opens the app on server start (default: on)
  • Notifications — desktop notifications (with an in-app toast fallback) for incoming friend requests and ready-to-install updates; opt in from the Notifications tab

Security

All peer-to-peer communication is encrypted at the application layer:

  • Key exchange — X25519 ECDH, with both sides signing via their Ed25519 identity key, so neither side can be impersonated
  • Session encryption — AES-256-GCM with a unique key and random IV per message, covering chat, search, friend requests, and file chunks
  • Identity — your node ID is the SHA-256 of your public key, so it's cryptographically bound to your keypair

The transport is plain ws://, not wss:// (TLS). A network observer can see which IPs are talking to each other, but cannot read the content of any messages. If you need to hide the fact of communication (not just its content), run Filenet behind a VPN or Tailscale.

The local database (SQLite) is not encrypted at rest. Anyone with filesystem access to the machine can read chat history, the file index, and friend lists.

Networking / port forwarding

There's no automatic NAT traversal — you forward one port yourself:

  1. Find your router's admin interface (usually 192.168.1.1 or 192.168.0.1)
  2. Look for "Port Forwarding" or "Virtual Server"
  3. Forward TCP port 7734 (or your configured port) to your machine's local IP
  4. Share your public IP address (or a domain name pointing to it) with friends

Forward only the P2P port. Never expose the web UI (port 3000) to the internet — it has no authentication and grants full control of the application, including browsing the host filesystem, to anyone who can reach it. It's meant for your home network only.

Post-download scripts

Scripts run in order after each download completes. Each one is a TypeScript/JavaScript file with a default export: 

import type { BunFile } from 'bun';

interface TransferStats {
  downloadTimeMs: number;
  bytesTransferred: number;
  maxSources: number;
}

export default async function ({ file, stats }: { file: BunFile; stats: TransferStats }) {
  console.log(`Downloaded ${file.name} in ${stats.downloadTimeMs}ms`);
}

Add script paths in Settings → Scripts and reorder them. If a script moves or renames the file, later scripts in the chain receive the updated path.
